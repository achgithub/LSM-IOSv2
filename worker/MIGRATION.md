v2 runs **two** regional shards — `uk` and `eu` — seeded by copying the
sports-data rows across from v1's per-league D1s. This was a one-time copy: v1's
databases were **not** touched, migrated, or repointed — read-only throughout.
v1 keeps running on its own per-league D1s exactly as before.

## Shard layout

| Shard | DB           | Leagues (v1 source DB)                                              |
| ----- | ------------ | ------------------------------------------------------------------ |
| `uk`  | `lsm-uk-db`  | PL (`lms-pl-db`), ELC (`lms-elc-db`)                               |
| `eu`  | `lsm-eu-db`  | PD (`lms-pd-db`), BL1 (`lms-bl1-db`), SA (`lms-sa-db`), FL1 (`lms-fl1-db`), DED (`lms-ded-db`) |

See `src/shards.ts` for the authoritative league→shard map.

## The one-off copy (`scripts/seed-from-v1.mjs`)

A plain `wrangler d1 export` won't work: v2 adds `league_id` to `fixtures`/
`standings`, which v1 doesn't have, so the exported INSERTs would violate v2's
NOT NULL. Instead `scripts/seed-from-v1.mjs` reads v1 rows as JSON and
regenerates INSERTs that stamp `league_id`, and seeds the new `leagues` rows
(replacing v1's baked `league.config.json`). It reads v1 **read-only** — it only
SELECTs from v1's DBs, never writes to them.

### Steps (run from `worker/`)

1. **Log in** (interactive — run yourself): `wrangler login`
2. **Create the shard databases**, then paste their ids into `wrangler.jsonc`
   (replacing the `PLACEHOLDER-*` values):
   ```
   wrangler d1 create lsm-uk-db
   wrangler d1 create lsm-eu-db
   ```
3. **Apply the v2 schema** to each shard:
   ```
   wrangler d1 execute lsm-uk-db --remote --file=./schema.sql
   wrangler d1 execute lsm-eu-db --remote --file=./schema.sql
   ```
4. **Dry run** — generates `_seed/lsm-uk-db.sql` / `_seed/lsm-eu-db.sql`, applies
   nothing. Inspect the SQL:
   ```
   node scripts/seed-from-v1.mjs
   ```
5. **Apply** — runs the generated SQL against the v2 shards:
   ```
   node scripts/seed-from-v1.mjs --apply
   ```
6. **Verify** row counts in each shard match v1 (per league), e.g.:
   ```
   wrangler d1 execute lsm-uk-db --remote --command \
     "SELECT league_id, COUNT(*) FROM fixtures GROUP BY league_id"
   ```

### What is NOT copied

- `attest_devices` — v2 re-attests on first launch under its own bundle id.
- `sync_meta` — regenerated by v2's first sync.
- `lms-demo-db` — the standalone screenshot/demo D1 stays a v1 concern.
- Any per-game/per-player data — v1 never had it (it lived on-device); v2's
  Layer-2 tables start empty.

## Full separation (after the copy is verified)

Once the shards hold the data and v2 reads its own copy, **cut every remaining
pointer from v2 to v1** so the two are fully independent:

1. **Delete the copier** — it is the only thing in v2 that names v1's DBs:
   ```
   git rm worker/scripts/seed-from-v1.mjs && rm -rf worker/_seed
   ```
2. **Neutralise the inherited v1 infra configs** carried over by the clone —
   `worker-registry/wrangler.jsonc` and `worker-dash/wrangler.jsonc` still hold
   v1's live D1 ids / `*.sportsmanager.site` routes. Either delete those two
   packages from the v2 tree (if v2 won't reuse them) or replace their live
   resource ids/routes with v2 placeholders, so no `wrangler deploy` from the v2
   clone can ever touch v1's production cloud. (Done as the separation step — see
   the repo root `V2-SCAFFOLD.md`.)
3. **Confirm** nothing v1 remains:
   ```
   grep -rn 'lms-pl-db\|lms-elc-db\|lms-pd-db\|sportsmanager.site' . \
     --include=*.jsonc --include=*.mjs --include=*.ts
   ```
   Expect no hits once separation is complete.
